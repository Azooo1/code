**3. JWT 密钥用的是占位符，且无启动校验**
`config.yaml` 里 `jwt.secret: your-super-secret-jwt-key-change-in-production`，而 `pkg/jwt` 直接从配置读，没有"检测到占位符就拒绝启动"的保护。这个值就印在这个源码包里——谁不改就能伪造任意 `user_id`（甚至管理员）的 token，直接接管任意账号。另外用户 token 和管理员 token 用的是**同一个密钥**（靠 issuer/claims 区分，目前能挡住，但属于不该共用）。**修复：启动时校验 secret 非默认值、足够长；用户/管理员分离密钥。**

## 🟠 高

**4. 登录无防爆破**
`auth_handler.go:442` 的 `Login` 除了全局 100 次/秒/IP 的限流外，没有任何按账号/IP 的登录失败锁定。设备锁短信验证（`issueDeviceLockChallenge`）默认是关的（注册时 `DeviceLockEnabled: false`）。已知用户名即可高速撞密码。**修复：登录失败计数 + 指数退避/锁定。**

**5. 上传可导致存储型 XSS**
`upload_handler.go` 把 `application/octet-stream` 当作合法图片类型放行（`allowedImageTypes`，本意是兼容 HEIC），而保存文件名的扩展名直接取自用户上传的原始文件名（`filepath.Ext(header.Filename)`），且 `/uploads` 用 `StaticFS` 提供服务（按扩展名自动设 Content-Type，`main.go:607`）。于是上传 `evil.html` 并把 `Content-Type` 设成 `application/octet-stream`，就会被存成 `xxx.html` 并以 `text/html` 在 API 同源下返回，触发存储型 XSS。**修复：扩展名按"已校验的真实类型"白名单生成，别信原文件名；去掉 octet-stream 兜底改为读字节嗅探；上传文件强制 `Content-Disposition: attachment` 或放独立域名。**

**6. 基础设施默认弱口令**
`config.yaml` / `baota/config.yaml`：MySQL `yiryanran/yiryanran`、Redis `123456`、MongoDB `grimimim/grimimim`。这些库若可被外网/内网横向触达就很危险。**修复：上线前全部改强密码并限制网络访问。**

## 🟡 中 / 加固建议

- **CORS 默认放开**：`allowed_origins` 为空时回退成 `*`（`middleware.go` 的 `allowedCORSOrigin`）。因为是 Bearer-token 而非 Cookie，CSRF 风险有限，但生产应填白名单。
- **手机号默认会被序列化**：`models/user.go:18` 的 `Phone` 带 `json:"phone"` 无脱敏。登录返回自己的信息没问题，但任何返回"别人 User 对象"的接口（联系人、群成员、搜索）都会泄露对方手机号——建议核查这些接口，统一脱敏或用专门的安全 DTO。
- **Token 生命周期过长**：访问 token 30 天、刷新 90 天，外加客户端为"抖音式切号"把 token 落本地（`account_manager.dart`）。一旦设备被取，影响面很大。建议缩短并支持服务端吊销（会话版本机制已经有了，是好的基础）。
- **`server.mode: debug`**：默认 debug，Gin 会输出详细错误页。生产改 `release`。
- **WS 用 query 传 token**：`/ws` 允许 `?token=`，可能被中间代理日志记录（它自己的访问日志已做脱敏，是亮点，但上游代理未必）。

## 🟢 做得好的地方（这套代码并非粗制滥造）

- **资金并发安全到位**：所有余额变更都用 `FOR UPDATE` 行锁 + `gorm.Expr("balance + ?")` 原子更新（充值/提现/红包/转账），没有典型的竞态双花。提现还做了支付密码校验、锁内复检余额、冻结后转管理员审核。
- **在线支付实现正确**：用官方 SDK 验签（微信 `NewRSANotifyHandler`、支付宝 `DecodeNotification`），`FulfillIfPaid` 幂等、校验金额一致、入账以订单金额为准——这块没问题。
- **JWT 固定 HMAC**，挡住了 `alg=none`/算法混淆；带会话版本可吊销。
- **口令都用 bcrypt**（登录密码 + 支付密码）。
- **E2EE 架构是真的**：按"每接收设备一个被包裹的内容密钥信封 + 密文/IV/MAC"建模（`models/message_e2ee.go`），服务端只存密文和被加密的密钥，不碰明文和私钥。
- **没发现后门**：扫了 Go/TS/Vue/Dart，没有硬编码万能口令、eval 投毒、隐藏管理账号；那两个 vendor 域名只出现在测试夹具里，不是"偷偷回传"。访问日志对 token/password 等做了脱敏。
